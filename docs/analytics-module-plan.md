# Life OPS 统计分析模块实施方案

> 基于当前代码库现状整理，时间点：2026-04-13
> 目标：为整个项目补齐一个“看全局、看现状、看成果、看未来进度”的统计分析模块

## 1. 先说结论

这个模块不应该直接堆在首页，也不应该做成传统 BI 式“大屏”。

更合适的做法是：

- 保持 `Home` 继续承担“一眼扫完，立即行动”的职责
- 新增一个独立的 **全局洞察页**，推荐命名为 `洞察`，路由建议为 `/insights`
- 首页只放 1-2 个轻量洞察入口卡片，把深度统计、趋势、成果、预测都放进洞察页

这样既符合当前项目在 `docs/00-vision/README.md` 里定义的三层结构，也不会破坏首页“低情绪负担”的产品原则。

## 2. 当前项目已经具备的可分析基础

从现有代码和数据表来看，项目已经不是“没数据”，而是“数据分散、没有统一视角”。

当前已经有这些真实可分析的功能模块：

| 领域/模块 | 当前能力 | 主要数据来源 |
|---|---|---|

| 待办 | 执行日、完成状态、优先级、逾期识别、日历分布 | `quick_notes` |
| 三只青蛙 | 当日最重要事项、完成情况 | `daily_frogs` |
| TIL | 每日学习记录 | `daily_til` |
| 健身 | 训练次数、总组数、训练量、连续训练、动作分布 | `workout_sessions` / `workout_sets` / `exercise_types` |
| 英语 | 查询、词卡、复习、词单、每日总结 | `english_queries` / `english_cards` / `english_daily_assignments` / `english_learning_logs` / `english_daily_summaries` |
| 成长项目 | 项目、Todo、成果/灵感记录、截止日期 | `growth_projects` / `project_todos` / `project_notes` |

### 2.2 目前已经存在的统计雏形

代码里其实已经有不少局部统计逻辑：

- 首页：青蛙完成数、TIL 条数、笔记条数、本周训练天数
- 健身：本周训练次数、总组数、总训练量、分类分布、连续训练天数
- 英语：今日查询、待复习、卡片总数、掌握数、词单完成情况
- 输出：总输出数、已发布数、草稿数
- 成长：各领域项目数、项目 Todo 完成率
- 家庭：待处理、逾期、本周完成
- 待办：今日/逾期/未安排/已完成统计、日历热区

所以这次不是从零做统计，而是把这些“局部统计”抽到一个统一分析层里。

## 3. 这个模块要回答的 5 个问题

建议把整个统计分析模块的产品目标收敛成 5 个问题：

1. 我现在整体状态怎么样？
2. 哪些领域在推进，哪些领域在偏离？
3. 最近已经产出了什么成果？
4. 如果按当前节奏继续，接下来会到什么进度？
5. 我下一步最应该补哪一块？

把这 5 个问题对应到界面上，模块就不会失控。

## 4. 模块定位与信息架构

## 4.1 模块命名建议

- 中文名称：`洞察`
- 英文内部名：`insights`
- 路由：`/insights`
- 侧边栏位置：建议放在 `主页` 下面、`待办` 上面

这样它是一个跨领域的分析视图，而不是某个具体业务 area。

## 4.2 页面结构

洞察页建议分成 6 个区域：

### A. 全局脉搏

用于一屏回答“整体状态如何”。

建议放 4-6 个全局卡片：

- 本周活跃领域数
- 本周完成动作总数
- 当前风险项数
- 最近 7 天成果数
- 节奏稳定度
- 当前最需要关注的领域

### B. 领域健康矩阵

把 `健身 / 英语 / 阅读 / AI / 输出 / 家庭 / 待办系统 / 财务` 放成统一卡片。

每张卡片只给 4 件事：

- 当前状态：`稳定 / 进行中 / 偏离 / 未接入`
- 本周期进度
- 最近成果
- 下一提醒

这会成为整个页面最核心的“总览区”。

### C. 成果看板

把“最近已经做成了什么”单独抽出来，避免页面只有风险感。

建议来源：

- `outputs.status = published`
- `project_notes.type = achievement`
- 英语掌握数增长
- 健身达到周目标
- 家庭本周完成任务
- 三只青蛙全完成日

### D. 偏离与风险

这个区域只放需要注意的事情，不要超过 5-7 条。

建议来源：

- 逾期待办
- 逾期家庭任务
- 临近截止但项目 Todo 完成率偏低
- 英语待复习积压
- 健身连续多天无训练
- 长时间无输出更新

### E. 趋势与预测

这是“未来进度”的核心区。

建议按 `7天 / 30天 / 本季度` 切换，展示：

- 行动力趋势
- 成果趋势
- 节奏趋势
- 预测完成情况

### F. 即将到来的节点

主要展示未来 7-14 天：

- 项目截止
- 家庭任务到期
- 本周训练目标剩余
- 英语词单/复习压力

## 5. 指标体系设计

## 5.1 指标分成 4 层

不要直接把所有计数都堆出来，建议统一为 4 层：

| 层级 | 作用 | 典型指标 |
|---|---|---|
| 节奏层 | 看连续性和稳定性 | 活跃天数、连续天数、周完成率 |
| 执行层 | 看是否在推进 | 新增数、完成数、完成率、未完成数 |
| 成果层 | 看有没有沉淀结果 | 已发布输出、成果记录、掌握词卡 |
| 风险层 | 看偏离和积压 | 逾期、待复习积压、临期项目、长时间无活动 |

## 5.2 全局总分建议

建议不要做复杂算法，第一版用一个可解释的健康分即可：

`health_score = 节奏分 30 + 执行分 35 + 成果分 20 - 风险扣分 15`

每个领域都能映射成 0-100 的健康分，再转成状态标签：

- `80-100`：稳定
- `60-79`：进行中
- `40-59`：需关注
- `<40`：偏离
- `未接入数据`：不参与评分，显示灰态

重点不是分数本身，而是让用户一眼知道“哪块稳、哪块需要拉回来”。

## 5.3 结合当前模块的指标映射

### 待办系统

- 当前待执行数
- 今日待办完成数
- 逾期数
- 未安排数
- 近 7 天完成率

### 三只青蛙

- 今日完成率
- 近 7 天全完成天数
- 连续完成天数

### TIL

- 今日记录数
- 近 7 天记录天数
- 分类分布

### 健身

- 本周训练天数 / 目标值
- 本周总组数
- 本周训练量
- 连续训练天数
- 最近 30 天训练频次趋势

### 英语

- 今日查询数
- 今日词单完成率
- 待复习数
- 掌握卡片数
- 最近 7/30 天学习活跃天数
- 新词积累速度

### 成长项目（阅读/AI/英语项目）

- 活跃项目数
- 项目总数
- Todo 完成率
- 临近截止项目数
- 最近成果记录数

### 输出

- 总输出数
- 已发布数
- 草稿数
- 最近 30 天发布数
- 草稿转发布率

### 家庭

- 待处理数
- 进行中数
- 本周完成数
- 逾期数
- 成员分工分布

### 财务

- 第一阶段不显示数值，只显示 `未接入`

这点很重要：没有数据时不能显示 `0`，否则会误导为“真的没有支出/资产/预算”，而不是“功能未接入”。

## 6. “现状 / 成果 / 未来进度”三条主线怎么落

用户这次明确提到的三件事，其实可以直接变成页面上的三类卡组。

## 6.1 现状

现状看的是“今天/本周现在是什么样”。

建议看：

- 领域健康状态
- 当前待处理数量
- 本周期进度
- 是否有明显偏离

## 6.2 成果

成果看的是“最近做成了什么，而不是做了多少动作”。

建议看：

- 已发布输出
- 项目成果记录
- 英语掌握卡片增长
- 健身达标周数
- 家庭本周完成事项

## 6.3 未来进度

未来进度看的是“按当前节奏继续走，大概率会走到哪里”。

建议用简单预测，不追求机器学习：

- 本周目标预计完成度
- 项目按当前 Todo 完成速度预计完成时间
- 最近 7 天输出节奏推算本月产出
- 英语按当前词单完成速度推算本周完成量

第一版可以全部用线性预测：

`预测值 = 已完成值 / 已过去时间 * 总周期时间`

这已经足够实用，也便于解释。

## 7. 数据架构建议

当前项目是 `Next.js 静态导出 + CSR + Supabase`，所以统计模块不要做成前端到处散查表。

最合适的方式是：

- 前端：只请求“分析视图”
- Supabase：负责聚合、统一口径、输出结果

## 7.1 建议新增的数据层

第一阶段建议新增：

### 1. `analytics_targets`

用来统一管理目标值和提醒阈值。

建议字段：

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `area_key` | 领域键，如 `fitness` / `english` / `output` |
| `metric_key` | 指标键，如 `weekly_workouts` |
| `period_type` | `day / week / month / quarter` |
| `target_value` | 目标值 |
| `warning_threshold` | 预警阈值 |
| `is_active` | 是否启用 |
| `updated_at` | 更新时间 |

原因很简单：现在很多目标是散落在代码里的，例如健身目标 `3` 天、英语日词单 `8` 个，这不利于后续统一分析。

### 2. `v_analytics_area_daily`

按“天 + 领域”聚合的视图。

建议字段：

- `stat_date`
- `area_key`
- `activity_count`
- `completed_count`
- `achievement_count`
- `risk_count`
- `active_flag`

### 3. `v_analytics_area_snapshot`

按“当前领域”输出洞察卡片数据。

建议字段：

- `area_key`
- `health_score`
- `status_label`
- `period_progress`
- `achievement_summary`
- `risk_summary`
- `next_focus`
- `updated_at`

### 4. `v_analytics_global_snapshot`

提供全局概览数据。

建议字段：

- `active_areas`
- `healthy_areas`
- `total_actions_this_week`
- `total_achievements_this_week`
- `risk_items`
- `focus_area`

### 5. `v_analytics_upcoming_items`

用于未来节点提醒。

建议整合：

- 项目截止日期
- 家庭任务到期
- 待办执行日期
- 未来几天的压力集中点

## 7.2 为什么优先用 View，而不是直接建很多新表

因为你现在已经有足够的数据明细表，问题主要不在“没有数据”，而在“没有统一汇总口径”。

所以第一阶段建议：

- 优先建 `view`
- 只有当性能或复杂度明显上来时，再引入 `materialized view`

这样改动最小，也最容易验证口径是否合理。

## 8.1 目录结构

建议新增一个完整 feature：

```text

    PeriodSwitcher.tsx
```

对应路由：

```text

```

## 8.2 组件职责建议

### `InsightsPage`

负责组合页面区块，不承担聚合逻辑。

### `GlobalPulse`

展示全局 KPI 条。

### `AreaHealthGrid`

展示各领域统一健康卡片，是页面主核心。

### `AchievementBoard`

展示最近成果，强调“已经做成的事情”。

### `RiskPanel`

展示少量高优先级偏离项。

### `TrendForecastPanel`

展示趋势图和预测卡。

### `UpcomingMilestones`

展示未来 7-14 天关键节点。

## 8.3 视觉设计建议

这个页面要克制，不要做成压迫感很强的监控面板。

建议：

- 默认看 `本周`
- 用卡片、条形趋势、小型热力视图，不用大面积复杂图表
- 风险区块最多显示前 5 条
- 成果区块必须和风险区块并列，保持心理平衡
- 财务未接入显示灰态说明，不显示 0
- 所有评分都要配解释文字，避免“黑箱感”

## 9. 分阶段实施路线

## Phase 1：先把“现状”做出来

目标：一版可用。

范围：

- 新增 `/insights`
- 建立全局快照 + 领域健康卡片
- 接入待办、健身、英语、成长项目、输出、家庭
- 财务显示未接入
- 默认周期只支持 `本周 / 近30天`

这一阶段重点是：

- 统一视角
- 统一口径
- 统一入口

不要急着做复杂预测。

## Phase 2：补齐“成果”和“未来进度”

范围：

- 增加成果看板
- 增加风险提示
- 增加未来节点区块
- 增加基于周期的简单预测
- 引入 `analytics_targets`

这一阶段做完，页面会真正回答：

- 现在怎样
- 最近做成了什么
- 按当前速度接下来会怎样

## Phase 3：做 AI 洞察总结

范围：

- 自动生成每周洞察摘要
- 自动提炼“偏离原因”和“下周建议”
- 支持周回顾/月回顾文案生成

但这一定要放在最后，因为前面先得把指标和口径固定下来。

## 10. 技术实现顺序建议

如果开始开发，我建议按这个顺序拆任务：

1. 新建 `analytics` feature 和 `/insights` 路由壳
2. 梳理并定义统一类型 `AnalyticsSnapshot`
3. 在 Supabase 新建第一批聚合 view
4. 打通前端 `analyticsApi`
5. 先落全局脉搏 + 领域健康矩阵
6. 再补成果、风险、未来节点
7. 最后加趋势和预测

这个顺序的好处是：

- 很快能看到一版页面
- 聚合口径先稳定
- 后续新增区块不会推翻前面结构

## 11. 当前代码库里需要特别注意的约束

## 11.1 Home 不要变成 BI 首页

`docs/00-vision/README.md` 已经明确：

- 首页不展示明细数据
- 首页不制造焦虑
- 首页强调下一步行动

所以统计分析页必须独立，不建议直接往首页硬塞多张图。

## 11.2 当前目标模型还不统一

这是你项目里做“未来进度”的最大短板。

目前：

- 健身目标值是前端常量
- 英语词单目标也是前端逻辑
- 输出暂无月目标
- 家庭暂无周期目标
- 财务还没接入

所以如果要认真做预测，必须尽快把目标配置从代码里抽出来。

## 11.3 当前部分统计是前端临时聚合

像健身、输出、成长、待办里，很多统计还是 React Query 拉明细后在前端计算。

洞察页不能沿用这套方式，否则：

- 查询会变多
- 页面会慢
- 口径会分散

所以分析模块必须有单独的数据聚合层。

---

## 12. 最终验收标准

这个模块上线时，我建议按下面标准验收：

1. 能在一个页面看见所有已接入领域的当前状态
2. 能明确区分“现状 / 成果 / 风险 / 未来节点”
3. 能看出至少 7 天和 30 天两个周期的变化
4. 财务等未接入领域不会误显示为 0
5. 首页保持简洁，只增加洞察入口，不被分析内容淹没
6. 所有统计都来自统一聚合口径，而不是页面各算各的
